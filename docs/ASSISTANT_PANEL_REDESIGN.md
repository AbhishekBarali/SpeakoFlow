# Assistant Panel Redesign & QoL Brief

> Hand-off brief for the implementing model. This document gives you **goals,
> product decisions, and verified facts** — it deliberately does **not** tell
> you how to implement. You are a capable engineer; you design the "how."
> Anything written as a specific behavior is a **requirement**, not a
> suggestion. Work **phase by phase**, stopping at each **GATE** for a human
> build + visual check before continuing.

---

## 0. Context

SpeakoFlow is a local-first voice app (Tauri 2 — Rust backend in `src-tauri/src`,
React + TypeScript frontend in `src`). It has two floating always-on-top windows:

- **STT / recording overlay** (`src/overlay/RecordingOverlay.tsx` + `.css`) — a
  tiny, dark, content-hugging pill shown while dictating. It is clean and
  minimal, and is the **visual north star** for this work.
- **Assistant panel** (`src/assistant/AssistantPanel.tsx` + `.css`) — a voice
  assistant window (voice → local STT → LLM → streaming answer, plus TTS, web
  search, screen vision). It is currently **too big, inconsistent, light by
  default, and janky.** This is what you are rebuilding.

**Mission:** make the Assistant window feel like the STT overlay's calmer
sibling — minimal, quiet, warm near-black glass with a light variant — while
keeping every feature and adding the QoL items below. When in doubt, choose the
smaller, quieter option. It should stay a **simple assistant**.

## 1. Hard constraints (do not violate)

1. **This is a goals brief, not a recipe.** Where behavior is specified, treat
   it as a requirement and implement it however you judge best. Do not wait to
   be told the steps.
2. **No Playwright / no automated screenshotting.** Do not add or rely on
   `src/assistant/shoot.mjs` or any headless-browser tooling. Final visual
   verification is manual (`bun run tauri dev`). A static preview page for
   eyeballing is fine; browser drivers are not.
3. **Use the graphify knowledge graph to learn the codebase** before editing:
   `graphify query "<question>"`, `graphify explain "<concept>"`,
   `graphify path "<A>" "<B>"` (graph in `graphify-out/`, plus `GRAPH_REPORT.md`).
4. **No new keyboard shortcuts.** The user already has many; add none. New
   capabilities are triggered from within the assistant UI.
5. **Clear error feedback is mandatory** (see §4). No silent failures, no
   endless spinners, no raw error dumps.
6. **i18n everything.** No hardcoded user-facing strings in JSX (ESLint enforces
   this). Add keys to `src/i18n/locales/en/translation.json` under `assistant` /
   `overlay`; other locales may fall back to English.
7. **Keep it green.** After each phase: `bun run build`, `bun run lint`, and if
   Rust changed, `cargo fmt` + `cargo clippy`. Fix everything you introduce.
8. **Never hand-edit `src/bindings.ts`** — it's generated by tauri-specta on
   build. Change commands/settings in Rust, rebuild, use the regenerated types.
9. **Backward-compatible settings.** When adding/removing `assistant_*` fields
   in `settings.rs`, use `#[serde(default)]` and tolerate legacy keys so existing
   stored settings still load. Never wipe user settings.
10. **Surgical `assistant.rs` edits.** It's large and central (pipeline + window
    management) — change only what a phase needs; don't refactor it wholesale.
11. **Preserve behavior** unless a phase says otherwise: TTS (local Kokoro),
    LLM providers, web search (Serper), screen vision, single-instance, CLI.
12. **Windows-first verification** (the user is on Windows). The windows use
    `backdrop-filter: blur()` and `force_panel_topmost`. Don't break macOS/Linux;
    note in your summary what you couldn't verify there.

## 2. Ground truth (facts you can rely on — so you don't re-derive or guess)

### The STT overlay = the design target

`src/overlay/RecordingOverlay.css`:

- Surface `rgba(24,22,20,0.94)`; border `1px solid rgba(255,255,255,0.08)`.
- Depth comes from inner highlights, **not** a drop shadow (the window is
  transparent): `inset 0 1px 0 rgba(255,255,255,0.07), inset 0 0 0 0.5px rgba(0,0,0,0.4)`.
- `backdrop-filter: blur(20px) saturate(1.4)`; `border-radius: 999px`.
- **Content-hugging:** `inline-flex; min-width: 46px; height: 30px` (36px locked).
- Font `"Plus Jakarta Sans"`; icon color `#f5f5f4`.
- **Waveform-led states:** recording = just the waveform (calm dots before the
  first audio frame); transcribing/processing = settled waveform + small
  `Loader2` spinner (no words). Cancel = hover-reveal `×`. Locked = inline
  `Cancel · waveform · Done(✓)`.

### Shared waveform component (reuse it — don't rebuild)

`<AudioWaveform levels={number[]} size="sm"|"md" barCount={n} active={bool} mode="reactive"|"shimmer"|"flow" />`
from `@/components/shared`.

### Current assistant window (what you're replacing)

- Frontend: `src/assistant/AssistantPanel.tsx`, `AssistantPanel.css`, `main.tsx`,
  `useKokoroTts.ts` (+ a preview harness — ignore/remove `shoot.mjs`).
- Rust: `src-tauri/src/assistant.rs` (window + pipeline),
  `commands/assistant.rs`, `settings.rs` (`assistant_*`), `overlay.rs` (STT sizing).
- Window label `assistant_panel`; starts **collapsed** (`PILL_MODE = true`).
  Pill is `232×64` (**too big**). Expanded presets: compact `340×440`,
  standard `420×560`, large `560×720`.
- **Position already persists** (`save_position` on `WindowEvent::Moved`, key
  `assistant_panel_position`) — keep it working.
- Screen vision is currently **one-shot** (`SCREEN_ARMED` + `take_screen_armed`
  consumes it each turn). "Sticky" means changing this to persist until toggled.
- `SCREENSHOT_MARKER = "[screenshot attached]"` is appended to stored user
  messages and stripped for display — keep both sides in sync
  (`assistant.rs` ↔ `AssistantPanel.tsx`).
- Non-vision-provider detection exists (`is_vision_unsupported_error`) — reuse it.

### The opacity bug

In `AssistantPanel.css`:
`--panel-surface: rgb(253 252 250 / calc(0.97 * var(--panel-alpha)))`. The `0.97`
cap means the surface is **never fully opaque**, so even at "100%" the transparent
window shows through. Max opacity must be **fully opaque**.

### Customization to REMOVE (product decision)

The 6 accent colors (`violet/blue/emerald/rose/amber/mono`), the accent "orb"
glow, the **opacity slider**, and the **size presets**. Keep only **light/dark**
(assistant-window scoped) and, if trivial, text size. Remove the matching
controls from the Panel Appearance settings + the corresponding settings fields
(backward-compatibly). Keep the settings **preview** sharing styles with the real
panel so they can't drift.

## 3. Design decisions (product requirements)

### 3.1 The small pill — a voice HUD (collapsed = default state)

- **Voice only.** No text input or search on the pill; typing lives in the
  expanded panel.
- **As small and calm as the STT overlay** (match its tokens in dark; a warm
  off-white variant in light). Content-hugging; height in the STT range
  (~30–40px), not a chunky box.
- **Waveform-led states** via `AudioWaveform`: idle (minimal rest), listening
  (reactive), thinking (shimmer — reuse an existing symbol, don't invent a new
  glyph), searching (globe + shimmer), speaking (speaker + flow).
- **Cancel by hovering the state to reveal `×`** — like the STT overlay. No
  separate stop/pause button.
- **Hands-free lock** shows inline `Cancel · wave · Done(✓)` (the ✓ finishes &
  sends), like the STT locked layout.
- **Sticky screen-vision:** when armed, a small **camera badge** is always
  visible; **hovering it disables it** in one click.
- **Closable without expanding.** Today the only close is via the big window —
  fix that. The user wants to close from the pill; a press-and-hold gesture is
  rejected (it breaks drag-to-position). Deliver a close affordance that doesn't
  fight dragging (hover-revealed control is a natural fit — but the requirement
  is the outcome, you choose the mechanism).
- Drag to move; position persists (already implemented — keep it).

### 3.2 The expanded panel — minimal but complete

- **One fixed, comfortable size** (no presets); draggable; position persists.
  **Nothing may clip** at that size in either theme.
- **Opacity bug fixed** (fully opaque at max).
- **Light + dark**, driven by the assistant theme setting. **No theme toggle in
  the panel header** — that control lives only in assistant settings.
- Keep every feature, laid out quietly: header (title/status, TTS toggle, clear,
  collapse-to-pill, close); scrollable markdown messages (keep `react-markdown` +
  `remark-gfm`, tables, code); input row (text, send/stop, web-search toggle,
  screen-vision toggle, attach image/file).
- Same visual language as STT: hairline rules, off-white ink, warm near-black in
  dark. No accent orb, no saturated chrome.

### 3.3 Motion & accessibility

- Subtle animation; honor `prefers-reduced-motion` (both windows already do).
- Preserve `role="status"` / `aria-label` on the pill and accessible names on
  all icon buttons (via i18n).

## 4. Error feedback (first-class requirement)

The user must never be confused about why something didn't work. Every failure a
user can hit must surface as a **short, plain-language message that says what
went wrong** and, where useful, what to do next. Never a silent no-op, a spinner
that never resolves, or a raw stack/error dump.

Cover at least:

- **Microphone** unavailable, in use, or permission denied.
- **No assistant model** configured or loaded (and it's needed).
- **LLM / provider** errors: auth/API-key, rate limit, network/timeout, and
  "this model can't accept images" (reuse `is_vision_unsupported_error`).
- **Web search** failure.
- **TTS** load/playback failure.
- **Screenshot / region capture** failure.

Rules:

- Show errors **where the user is looking**: the pill for the compact state, the
  panel for the expanded state (a compact pill error should be readable without
  forcing an expand).
- Distinguish **transient** (retry) from **blocking** (needs a fix) errors.
- Keep the wording calm and specific; localize it (i18n).
- Errors count toward "done" — a phase isn't complete if its failure paths are
  silent or cryptic.

## 5. Phases

Work in order. Each ends at a **GATE**: stop, run build + lint (+ cargo if Rust),
and let the human eyeball it in `bun run tauri dev` before continuing. Phase 6
may be a **separate session**.

### Phase 0 — Orientation (no code)

**Goal:** understand the current assistant window, the STT overlay, the window
sizing + pipeline in `assistant.rs`, the `assistant_*` settings, the screen-vision
path, and how `bindings.ts` regenerates — enough to confirm this brief against
reality.
**Done when:** you've produced a short architecture summary and flagged anything
here that conflicts with the code, and the human agrees before you edit.

### Phase 1 — Shared foundation, decluttered, opacity fixed

**Goal:** the assistant window shares one visual language with the STT overlay
(dark near-black glass + a light variant), with accents / orb / opacity-slider /
size-presets removed and the opacity bug fixed. The Panel Appearance settings and
its live preview reflect this and cannot drift from the real panel.
**Done when:** panel and preview match; light and dark are both correct; max
opacity is truly opaque; only light/dark (+ optional text size) remain;
pre-existing stored settings still load without error.

### Phase 2 — The small voice pill

**Goal:** deliver §3.1 — a voice-only pill as small and calm as the STT overlay,
state carried by the waveform, hover-to-cancel, idle-hover to expand/close,
hands-free lock with cancel+done, and a sticky screen-vision badge you can
disable on hover.
**Done when:** the pill is markedly smaller; every state is legible in both
themes; you can cancel an in-flight turn and close the pill **without** expanding;
drag + position persistence still work; nothing clips.

### Phase 3 — The expanded panel

**Goal:** deliver §3.2 — one fixed, comfortable, non-clipping chat window in the
same language, all features present and quiet, opacity fixed, both themes, theme
control only in settings.
**Done when:** all existing features work; collapse/expand round-trips cleanly
with the pill; position persists; nothing clips in either theme.

### Phase 4 — See things: images, files, and focused screen vision

**Goal:** make the expanded panel able to chat about visual/document context:

- **Attach images** (button, paste, and drag-drop) and **text/pdf/code files** as
  context.
- **Region screenshot:** the screen-vision capture should let the user **draw a
  box to screenshot just part of the screen** and chat about that region, with
  full-screen as a fallback — triggered from the assistant's existing capture
  control, no new hotkey. (Builds on the existing screenshot pipeline —
  `screenshot.rs` + the image assembly in `assistant.rs`.)
- Screen vision is **sticky** and clearly indicated (per §3.1).
  Reuse the existing image pipeline and respect its size budget; handle non-vision
  providers per §4.
  **Done when:** you can attach/paste/drag an image or file and get a grounded
  answer; you can snip a region and ask about it; sticky vision persists across
  turns and is visible; oversized inputs and unsupported providers produce clear
  errors (per §4).

### Phase 5 — Conversation quality-of-life

**Goal:** **Regenerate** and **Continue** on the last answer; a **/summarize**
action that keeps long chats coherent (the only slash-command — don't build a
broader system); **per-code-block copy** (keep copy-whole-answer too).
**Done when:** regenerate/continue work on the latest turn, summarize keeps a
long conversation usable, and code blocks are individually copyable.

### Phase 6 — Resume past conversations from History _(may be a separate session)_

**Goal:** close the gap where assistant history exists but you can't reopen it —
select a past conversation in History and resume it in the panel. (Relevant code:
`managers/history.rs`, the conversation state in `assistant.rs`,
`assistantGetConversation`.)
**Done when:** you pick an old conversation → it loads into the panel → you can
continue it.

## 6. Definition of done (every phase)

- `bun run build` passes; `bun run lint` clean; `cargo fmt` + `cargo clippy` clean
  if Rust changed; `bindings.ts` regenerated (not hand-edited).
- All new strings localized; no hardcoded JSX strings.
- **Failure paths give clear, localized feedback** (§4) — not silent or cryptic.
- A written self-review: pill + expanded, **light and dark**, each state
  (idle / listening / thinking / searching / speaking / locked / error) — nothing
  clips, alignment clean, motion respects reduced-motion.
- A note on what was verified on Windows and what couldn't be checked elsewhere.
- No regressions in TTS, providers, web search, screen vision, CLI, single-instance.
